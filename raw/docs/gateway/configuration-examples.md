以下の例は現在の設定スキーマに沿っています。網羅的なリファレンスとフィールドごとの注記については、 [設定](https://docs.openclaw.ai/ja-JP/gateway/configuration) を参照してください。

## クイックスタート

### 絶対最小構成

json5

```
{
  agents: { defaults: { workspace: "~/.openclaw/workspace" } },
  channels: { whatsapp: { allowFrom: ["+15555550123"] } },
}
```

`~/.openclaw/openclaw.json` に保存すると、その番号からボットにダイレクトメッセージを送れます。

### 推奨スターター

json5

```
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace",
      model: { primary: "anthropic/claude-sonnet-4-6" },
    },
    list: [
      {
        id: "main",
        identity: {
          name: "Clawd",
          theme: "helpful assistant",
          emoji: "🦞",
        },
      },
    ],
  },
  channels: {
    whatsapp: {
      allowFrom: ["+15555550123"],
      groups: { "*": { requireMention: true } },
    },
  },
  messages: {
    visibleReplies: "automatic",
    groupChat: {
      visibleReplies: "message_tool", // default; use "automatic" for legacy room replies
    },
  },
}
```

### シンボリックリンクされた兄弟スキルリポジトリ

組み込みスキルルートに兄弟リポジトリへのシンボリックリンクが含まれる場合に使用します。たとえば `~/.agents/skills/manager -> ~/Projects/manager/skills` です。

json5

```
{
  skills: {
    load: {
      extraDirs: ["~/Projects/manager/skills"],
      allowSymlinkTargets: ["~/Projects/manager/skills"],
    },
  },
}
```
- `extraDirs` は、兄弟リポジトリを明示的なスキルルートとしてスキャンします。
- `allowSymlinkTargets` は、任意のシンボリックリンク脱出を許可せずに、シンボリックリンクされたスキルフォルダーをその信頼済みの実ターゲットルートへ解決できるようにします。

## 一般的なパターン

### 共有スキルベースラインと 1 つの上書き

json5

```
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace",
      skills: ["github", "weather"],
    },
    list: [
      { id: "main", default: true },
      { id: "docs", workspace: "~/.openclaw/workspace-docs", skills: ["docs-search"] },
    ],
  },
}
```
- `agents.defaults.skills` は共有ベースラインです。
- `agents.list[].skills` は、1 つのエージェントについてそのベースラインを置き換えます。
- エージェントにスキルを見せない場合は `skills: []` を使用します。

### マルチプラットフォーム設定

json5

```
{
  agents: { defaults: { workspace: "~/.openclaw/workspace" } },
  channels: {
    whatsapp: { allowFrom: ["+15555550123"] },
    telegram: {
      enabled: true,
      botToken: "YOUR_TOKEN",
      allowFrom: ["123456789"],
    },
    discord: {
      enabled: true,
      token: "YOUR_TOKEN",
      dm: { allowFrom: ["123456789012345678"] },
    },
  },
}
```

### 信頼済みノードネットワークの自動承認

ネットワーク経路を制御している場合を除き、デバイスのペアリングは手動のままにしてください。専用の ラボまたは tailnet サブネットでは、正確な CIDR または IP を指定して、初回のノードデバイス自動承認を 有効にできます。

json5

```
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24", "fd00:1234:5678::/64"],
      },
    },
  },
}
```

未設定の場合、これは無効のままです。これは、要求されたスコープがない新規の `role: node` ペアリングにのみ 適用されます。オペレーター/ブラウザークライアント、およびロール、スコープ、メタデータ、または 公開鍵のアップグレードでは、引き続き手動承認が必要です。

### セキュアDMモード（共有受信箱 / マルチユーザーDM）

複数の人がボットにDMできる場合（ `allowFrom` に複数のエントリがある、複数人のペアリング承認がある、または `dmPolicy: "open"` ）、 **セキュアDMモード** を有効にして、異なる送信者からのDMがデフォルトで1つのコンテキストを共有しないようにします。

json5

```
{
  // Secure DM mode (recommended for multi-user or sensitive DM agents)
  session: { dmScope: "per-channel-peer" },
 
  channels: {
    // Example: WhatsApp multi-user inbox
    whatsapp: {
      dmPolicy: "allowlist",
      allowFrom: ["+15555550123", "+15555550124"],
    },
 
    // Example: Discord multi-user inbox
    discord: {
      enabled: true,
      token: "YOUR_DISCORD_BOT_TOKEN",
      dm: { enabled: true, allowFrom: ["123456789012345678", "987654321098765432"] },
    },
  },
}
```

Discord/Slack/Google Chat/Microsoft Teams/Mattermost/IRC では、送信者の認可はデフォルトでID優先です。 そのリスクを明示的に受け入れる場合に限り、各チャンネルの `dangerouslyAllowNameMatching: true` で、直接変更可能な名前/メール/ニックネームの照合を有効にしてください。

## ヒント

- `dmPolicy: "open"` を設定する場合、対応する `allowFrom` リストには `"*"` を含める必要があります。
- プロバイダーIDはそれぞれ異なります（電話番号、ユーザーID、チャンネルID）。形式を確認するにはプロバイダーのドキュメントを使用してください。
- 後で追加できる任意セクション: `web` 、 `browser` 、 `ui` 、 `discovery` 、 `plugins` 、 `talk` 、 `signal` 、 `imessage` 。
- より詳しいセットアップメモについては、 [プロバイダー](https://docs.openclaw.ai/ja-JP/providers) と [トラブルシューティング](https://docs.openclaw.ai/ja-JP/gateway/troubleshooting) を参照してください。
