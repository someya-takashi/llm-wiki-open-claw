# 長時間実行エージェントのための効果的なハーネス

> 原典: `raw/articles/Effective harnesses for long-running agents.md` ・ https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents
> 要約は [[articles/effective-harnesses-for-long-running-agents]]

AI エージェントがより高性能になるにつれ、開発者はますます、数時間、あるいは数日に及ぶ作業を要する複雑なタスクをエージェントに任せるよう求めるようになっています。しかし、複数のコンテキストウィンドウをまたいでエージェントに一貫した進捗を出させることは、依然として未解決の問題のままです。

長時間実行エージェントの中核的な課題は、それらが離散的なセッションで作業しなければならず、各新規セッションが、それ以前に何があったかの記憶なしに始まることです。シフト制で働くエンジニアによって人員配置されたソフトウェアプロジェクトを想像してください。そこでは各新規エンジニアが、前のシフトで何が起きたかの記憶なしに到着します。コンテキストウィンドウは限られており、ほとんどの複雑なプロジェクトは単一のウィンドウ内で完了できないため、エージェントはコーディングセッションの間のギャップを埋める方法を必要とします。

私たちは、[Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) が多数のコンテキストウィンドウにわたって効果的に作業できるようにする二段構えの解決策を開発しました：初回実行時に環境をセットアップする**初期化エージェント（initializer agent）**と、毎セッションで漸進的な進捗を出しつつ次のセッションのために明確な成果物（artifacts）を残すことを任務とする**コーディングエージェント（coding agent）**です。コード例は付属の [クイックスタート](https://github.com/anthropics/claude-quickstarts/tree/main/autonomous-coding) にあります。

## 長時間実行エージェントの問題

Claude Agent SDK は、コーディングに長けた強力で汎用的なエージェントハーネスであり、モデルがツールを使ってコンテキストを集め・計画し・実行することを要する他のタスクにも長けています。これはコンパクション（compaction）のようなコンテキスト管理機能を持ち、これによりエージェントはコンテキストウィンドウを使い果たすことなくタスクに取り組めます。理論上は、このセットアップを踏まえれば、エージェントが任意に長い時間、有用な作業を続けることが可能なはずです。

しかし、コンパクションだけでは不十分です。そのままでは、Claude Agent SDK 上でループ内で複数のコンテキストウィンドウにわたって動作する Opus 4.5 のようなフロンティアのコーディングモデルでさえ、「[claude.ai](http://claude.ai/redirect/website.v1.a041fd6f-639f-4713-834e-8f6ee942ad6b) のクローンを作れ」のような高レベルなプロンプトだけを与えられた場合、本番品質の Web アプリの構築には及びません。

Claude の失敗は 2 つのパターンで現れました。第一に、エージェントは一度にやりすぎようとする傾向がありました――本質的にアプリをワンショット（一発）で作ろうとするのです。しばしばこれは、モデルが実装の途中でコンテキストを使い果たし、次のセッションが機能を半分実装・未文書化のまま始まる事態を招きました。するとエージェントは何が起きたかを推測しなければならず、基本的なアプリを再び動かそうとするのに多大な時間を費やすことになります。これはコンパクションがあっても起こります。コンパクションは、次のエージェントへ常に完璧に明確な指示を渡すわけではないからです。

第二の失敗モードは、プロジェクトの後半でしばしば発生しました。いくつかの機能がすでに構築された後、後続のエージェントインスタンスが周囲を見回し、進捗が出ていることを見て、仕事は完了したと宣言するのです。

これは問題を 2 つの部分に分解します。第一に、与えられたプロンプトが要求する*すべて*の機能の基礎を据える初期環境をセットアップする必要があり、それがエージェントを段階的に・機能ごとに作業するよう仕向けます。第二に、各エージェントに、目標に向けて漸進的な進捗を出しつつ、セッションの終わりに環境をきれいな状態（clean state）に残すよう促すべきです。「きれいな状態」とは、メインブランチへのマージにふさわしい種類のコードを意味します：重大なバグがなく、コードは整然として十分に文書化され、概して開発者が、無関係な混乱をまず片付ける必要なしに新機能の作業を容易に始められる状態です。

社内で実験する中で、私たちはこれらの問題を二部構成の解決策で扱いました：

1. 初期化エージェント（Initializer agent）：最初のエージェントセッションは、モデルに初期環境のセットアップを求める専用プロンプトを使います：`init.sh` スクリプト、エージェントが何をしたかのログを保つ claude-progress.txt ファイル、そしてどのファイルが追加されたかを示す初回の git コミットです。
2. コーディングエージェント（Coding agent）：以降のすべてのセッションは、モデルに漸進的な進捗を出させ、その後に構造化された更新を残させます。<sup>1</sup>

ここでの重要な洞察は、新しいコンテキストウィンドウで始めるときにエージェントが作業の状態を素早く理解する方法を見つけることであり、これは git 履歴と並ぶ claude-progress.txt ファイルによって達成されます。これらの実践のインスピレーションは、有能なソフトウェアエンジニアが毎日していることを知ることから得られました。

## 環境管理

更新された [Claude 4 プロンプティングガイド](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices#multi-context-window-workflows) では、「最初のコンテキストウィンドウのための異なるプロンプト」を使うハーネス構造を含む、複数コンテキストウィンドウのワークフローのためのベストプラクティスをいくつか共有しました。この「異なるプロンプト」は、将来のコーディングエージェントが効果的に作業するために必要となるすべてのコンテキストを備えた環境を、初期化エージェントにセットアップするよう要求します。ここでは、そのような環境の主要な構成要素のいくつかについて、より深く掘り下げます。

### 機能リスト（Feature list）

エージェントがアプリをワンショットで作ったり、プロジェクトを早まって完了とみなしたりする問題に対処するため、私たちは初期化エージェントに、ユーザーの初期プロンプトを敷衍した包括的な機能要件ファイルを書くよう促しました。[claude.ai](http://claude.ai/redirect/website.v1.a041fd6f-639f-4713-834e-8f6ee942ad6b) クローンの例では、これは「ユーザーが新しいチャットを開き、クエリを入力し、Enter を押し、AI の応答を見られる」のような 200 を超える機能を意味しました。これらの機能はすべて当初「失敗（failing）」と印付けされ、後続のコーディングエージェントが完全な機能性がどう見えるかの明確な概要を持てるようにしました。

```
{
    "category": "functional",
    "description": "New chat button creates a fresh conversation",
    "steps": [
      "Navigate to main interface",
      "Click the 'New Chat' button",
      "Verify a new conversation is created",
      "Check that chat area shows welcome state",
      "Verify conversation appears in sidebar"
    ],
    "passes": false
  }
```

私たちはコーディングエージェントに、このファイルを `passes` フィールドのステータスを変えることによってのみ編集するよう促し、「テストを削除または編集することは受け入れられない。なぜならそれは機能の欠落やバグにつながりうるからだ」のような強い言葉の指示を使います。いくらかの実験の後、私たちはこれに JSON を使うことに落ち着きました。モデルは Markdown ファイルと比べて JSON ファイルを不適切に変更・上書きする可能性が低いからです。

### 漸進的な進捗（Incremental progress）

この初期環境の足場を踏まえ、コーディングエージェントの次の反復には、一度に 1 つの機能だけに取り組むよう求めました。この漸進的アプローチは、一度にやりすぎるエージェントの傾向に対処するうえで決定的であることが判明しました。

漸進的に作業するようになっても、モデルがコード変更を行った後に環境をきれいな状態に残すことは依然として不可欠です。私たちの実験では、この挙動を引き出す最良の方法は、モデルに記述的なコミットメッセージで進捗を git にコミットさせ、進捗ファイルに進捗の要約を書かせることだと分かりました。これによりモデルは git を使って悪いコード変更を取り消し、コードベースの動作する状態を復元できました。

これらのアプローチは効率も高めました。エージェントが何が起きたかを推測し、基本的なアプリを再び動かそうとするのに時間を費やす必要をなくしたからです。

### テスト（Testing）

私たちが観察したもう 1 つの主要な失敗モードは、適切なテストなしに機能を完了と印付けする Claude の傾向でした。明示的なプロンプトがないと、Claude はコード変更を行い、開発サーバーに対してユニットテストや `curl` コマンドでテストさえするものの、機能がエンドツーエンドで動作しないことを認識し損ねる傾向がありました。

Web アプリ構築のケースでは、Claude はブラウザ自動化ツールを使い、人間のユーザーがするようにすべてのテストを行うよう明示的に促されると、おおむね機能をエンドツーエンドで検証することがうまくできました。

Claude にこの種のテストツールを提供することは、性能を劇的に改善しました。エージェントがコードだけからは明白でなかったバグを識別し修正できたからです。

いくつかの問題は残っています。例えば、Claude の視覚（vision）とブラウザ自動化ツールの限界が、あらゆる種類のバグの識別を難しくしています。例えば、Claude は Puppeteer MCP を通じてブラウザネイティブのアラートモーダルを見ることができず、その結果、これらのモーダルに依存する機能はバグを含みやすい傾向がありました。

## 状況把握（Getting up to speed）

以上のすべてが整った状態で、すべてのコーディングエージェントは、状況を把握するための一連のステップ――いくつかはごく基本的だが依然として有用――を踏むよう促されます：

1. *`pwd` を実行して作業中のディレクトリを確認する。あなたはこのディレクトリ内のファイルしか編集できない。*
2. *git ログと進捗ファイルを読んで、最近何が作業されたかの状況を把握する。*
3. *機能リストファイルを読み、まだ完了していない最優先の機能を選んで取り組む。*

このアプローチは、Claude がコードをテストする方法を考え出す必要がないため、毎セッションでトークンをいくらか節約します。また、初期化エージェントに開発サーバーを実行できる init.sh スクリプトを書かせ、それから新機能を実装する前に基本的なエンドツーエンドテストを走らせることも役立ちます。

claude.ai クローンのケースでは、これはエージェントが常にローカル開発サーバーを起動し、Puppeteer MCP を使って新しいチャットを開始し、メッセージを送り、応答を受け取ることを意味しました。これにより Claude は、アプリが壊れた状態に残されていないかを素早く識別し、既存のバグを即座に修正できました。もしエージェントが代わりに新機能の実装を始めていたら、おそらく問題を悪化させていたでしょう。

これらすべてを踏まえると、典型的なセッションは次のようなアシスタントメッセージで始まります：

```
[Assistant] I'll start by getting my bearings and understanding the current state of the project.
[Tool Use] <bash - pwd>
[Tool Use] <read - claude-progress.txt>
[Tool Use] <read - feature_list.json>
[Assistant] Let me check the git log to see recent work.
[Tool Use] <bash - git log --oneline -20>
[Assistant] Now let me check if there's an init.sh script to restart the servers.
<Starts the development server>
[Assistant] Excellent! Now let me navigate to the application and verify that some fundamental features are still working.
<Tests basic functionality>
[Assistant] Based on my verification testing, I can see that the fundamental functionality is working well. The core chat features, theme switching, conversation loading, and error handling are all functioning correctly. Now let me review the tests.json file more comprehensively to understand what needs to be implemented next.
<Starts work on a new feature>
```

エージェントの失敗モードと解決策

| **問題** | **初期化エージェントの挙動** | **コーディングエージェントの挙動** |
| --- | --- | --- |
| Claude がプロジェクト全体の勝利を早まって宣言する。 | 機能リストファイルをセットアップする：入力仕様に基づき、エンドツーエンドの機能記述のリストを持つ構造化 JSON ファイルをセットアップする。 | セッションの冒頭で機能リストファイルを読む。取り組む単一の機能を選ぶ。 |
| Claude が環境をバグや未文書化の進捗のある状態に残す。 | 初回の git リポジトリと進捗ノートファイルが書かれる。 | 進捗ノートファイルと git コミットログを読んでセッションを始め、開発サーバーで基本テストを走らせて未文書化のバグを捕まえる。git コミットと進捗更新を書いてセッションを終える。 |

| Claude がアプリの実行方法を考え出すのに時間を費やす。 | 開発サーバーを実行できる `init.sh` スクリプトを書く。 | `init.sh` を読んでセッションを始める。 |

## 今後の課題（Future work）

この研究は、モデルが多数のコンテキストウィンドウにわたって漸進的な進捗を出せるようにするための、長時間実行エージェントハーネスにおける 1 つの可能な解決策の集合を実証します。しかし、未解決の問いは残ります。

最も注目すべきは、単一の汎用コーディングエージェントがコンテキストをまたいで最も良く機能するのか、それともマルチエージェントアーキテクチャを通じてより良い性能が達成できるのかが、依然として不明なことです。テストエージェント、品質保証（QA）エージェント、コードクリーンアップエージェントのような専門化されたエージェントが、ソフトウェア開発ライフサイクル全体のサブタスクでさらに良い仕事をしうる、と考えるのは合理的に思えます。

加えて、このデモはフルスタックの Web アプリ開発に最適化されています。今後の方向性は、これらの知見を他の分野へ一般化することです。これらの教訓の一部またはすべてが、例えば科学研究や金融モデリングで要求される種類の長時間実行エージェントタスクに適用できる可能性が高いです。
